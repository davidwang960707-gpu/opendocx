# 安全策略

OpenDocX 仍处于 v0.1.0-alpha 阶段。当前版本适合本地试用、内部评估和非敏感场景验证，不建议直接承载生产敏感数据。

## 支持版本

| 版本 | 状态 |
| --- | --- |
| v0.1.x | 接受安全修复 |
| 更早版本 | 不支持 |

## 报告安全问题

如果你发现安全问题，请不要在公开 Issue 中披露细节。

建议邮件标题：

```text
[OpenDocX Security] 漏洞简述
```

报告中请尽量包含：

- 影响范围。
- 复现步骤。
- 预期结果与实际结果。
- 相关日志或截图。
- 你建议的修复方向。

## 当前安全边界

已覆盖：

- 密码哈希存储。
- JWT 登录认证。
- 管理员路由保护。
- Markdown 默认转义与清洗。
- 反馈限流和基础垃圾反馈判断。
- 审计日志。
- 开源仓不包含 `.env`、真实数据库、本地构建产物和内部数据。

仍需增强：

- Refresh token 与会话吊销。
- 更细粒度的项目权限。
- 媒体上传安全扫描。
- 静态站 CSP。
- 更完整的生产日志、告警和审计导出。

## 部署建议

如果你在自己的服务器部署：

- 修改默认管理员密码。
- 使用强 `JWT_SECRET`。
- 不要把真实 LLM API Key 提交到仓库。
- 只通过 HTTPS 暴露公网服务。
- 限制数据库端口访问。
- 定期备份数据库。

## API Key 轮换

如果怀疑 LLM API Key 泄露：

1. 立刻到服务商后台吊销旧 Key。
2. 生成新 Key。
3. 更新 `.env`。
4. 重启后端。
5. 验证 AI 编辑功能。

## 开源发布说明

公开版本只包含源码、Demo seed、中文文档、截图和必要配置示例。不包含真实运行数据、内部项目数据、历史仓库记录或个人路径。
