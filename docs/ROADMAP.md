# OpenDocX 路线图

OpenDocX 的目标不是成为另一个通用静态站生成器，而是成为 AI / Vibe Coding 项目的文档管理、项目手册和发布工作台。

## 产品定位

AI 可以很快生成代码，但项目真正交付前还缺这些东西：

- 清晰的项目手册。
- 可维护的文档树。
- 可发布的静态站。
- 读者反馈入口。
- 文档变更审计。
- 从草稿到发布的确认流程。

OpenDocX 聚焦这条最后一公里。

- 发布前确认。

## v0.2.0：开源体验增强

目标：让第一次接触项目的人 10 分钟内跑起来，并能看懂核心价值。

计划：

- 完善 README 和截图。
- 增加更多中文 Demo 文档。
- 增强静态站首页、目录、阅读进度和移动端体验。
- 优化审计日志、反馈列表的动态列宽和筛选。
- 增加基础前端测试。
- 提供公开演示站。

## v0.3.0：Markdown 与媒体能力

目标：让 OpenDocX 可以承载更真实的产品文档。

已完成基础能力：

- 图片上传、选择、复用和 Markdown 插入。
- GIF、视频和常见附件上传到版本资产库。
- 静态站构建时复制资产并重写引用路径。
- 右侧文档 Insight 使用规则分析术语、接口、摘要、健康度和相关文档。

计划：

- 资产库搜索、筛选、重命名和批量删除。
- 粘贴图片自动上传。
- 视频封面、体积提示和更友好的移动端展示。
- 公式完整渲染。
- Mermaid 完整渲染。
- 更丰富的提示块和代码块主题。
- 静态站 SEO、站点地图和元信息。

## v0.4.0：团队协作

目标：从个人工具走向小团队工作台。

计划：

- 项目成员权限。
- 文档审批流。
- 发布权限控制。
- 版本差异对比。
- 文档变更记录。
- 审计日志导出。

## v0.5.0：AI 工作流深化

目标：让 AI 不只是“写一句”，而是参与文档质量提升。

计划：

- 项目上下文感知写作。
- 文档质量评分。
- 缺失章节建议。
- 术语一致性检查。
- 从代码/API 自动生成文档草稿。
- 生成 OpenAPI 规范、SDK、测试用例等工程化文档动作。
- 面向静态站读者的 AI 问答入口。

## v1.0.0：稳定版本

目标：

- API 稳定。
- 测试覆盖关键路径。
- 生产部署文档完整。
- 多用户权限可用于真实团队。
- 静态站阅读体验接近成熟文档框架。

## 与 Docusaurus 的关系

Docusaurus 是成熟的文件型静态站生成器。OpenDocX 不打算在 v0.x 阶段硬碰它的生态，而是走不同方向：

- Docusaurus 适合已经有 Git 文档流程的团队。
- OpenDocX 适合需要 Web 管理后台、AI 辅助、发布确认、反馈和审计的项目。
